Model

The Model represents the data and business rules of the application.

  • Talks to the database
  • Validates data
  • Contains business logic
  • Knows how data can be created, changed and queried

In Rails, models are Ruby classes that usually map to database tables.

Think rules and data.

View

The View is responsible for what the user sees.

  • Displays data
  • Contains HTML templates
  • May include simple logic for presentation
  • Does not make decisions or change data

In Rails, views are templates that generate HTML.

Think display.

Controller

The Controller coordinates between the model and the view.

  • Receives requests from the browser
  • Asks models for data
  • Sends data to views
  • Decides which view to render

Controllers should not contain business logic.

Think traffic controller.

MVC

Model View Controller (MVC) is a way of structuring an application so that responsibilities are clearly separated.

Each part has a single job, which makes the system easier to understand, test, and change.

How MVC works in practice

  1. The user interacts with the View

    For example, selecting a customer to view their orders.

  2. The View sends a request to the Controller

    The View does not fetch data itself.

    It asks the Controller what it needs.

  3. The Controller asks the Model for data

    The Controller coordinates the request and calls the appropriate model methods.

  4. The Model returns the data

    The Model handles business rules and data access, such as retrieving orders and calculating totals.

  5. The View displays the result

    The Controller passes the data back to the View, which displays it to the user.

Key ideas to remember

  • The View displays data only
  • The Controller coordinates actions and flow
  • The Model contains rules, data, and calculations
  • The View never talks directly to the Model
  • The Model never knows about the View
